## Surprises and risks

### 1. `generate-api-docs.ts` — double `..` bug after T3 (HIGH)

`REPO_ROOT = path.resolve(process.cwd(), '..')` currently works because npm workspace scripts for `backend` set cwd = `backend/`, so `..` = repo root. After T3 the workspace is `packages/api`, so `..` = `packages/` — one directory short. The generator would write `api-manifest.json` and `API.md` into `packages/docs/` (which doesn't exist), silently failing. **Fix: change `'..'` to `'..', '..'` on line 43.**

### 2. nginx `root` is a hard-coded absolute path on the production host (HIGH)

`/home/rw3iss/sites/rw/frontend/dist` must be updated in `rw.ryanweiss.net.nginx.conf` AND on the production server's live nginx config. The file in the repo controls the reference, but nginx on the server needs `nginx -s reload` after the deploy that rebuilds into `packages/cms/dist`. If deploy runs before the nginx config is reloaded the site goes blank. **Recommend: include `nginx -s reload` (or `systemctl reload nginx`) in deploy.sh after a successful build.**

### 3. pm2 `--cwd` hard-codes `backend/` path — affects all cwd-relative paths (HIGH)

`deploy.sh:71: --cwd "$PROJECT_ROOT/backend"` controls `process.cwd()` for the live process. This governs: upload dir (`./uploads`), data/avatar dir (`./data`), SSR static-html cache (`cache/static-html`), `.env` lookup, and the `../frontend/dist` SSR dist resolution. All of these shift from `backend/…` to `packages/api/…` atomically when the `--cwd` line is updated. **The existing data directories (`backend/uploads/`, `backend/data/`) must be migrated to `packages/api/` on the production host, or `UPLOAD_DIR` and `DATA_DIR` env vars must be set to absolute paths in `.env` before the first deploy post-move.**

### 4. Docker volume `./uploads:/app/uploads` is anchored to compose-file location (MEDIUM)

In the current configuration compose lives at repo root, so `./uploads` = repo root. The `WORKDIR /app` plus `process.cwd()` = `/app` (npm workspace doesn't cd into the subpackage in Docker) means the container's upload dir is `/app/uploads`, matching the volume. **No code change needed for T3.** However, if T4 moves `docker-compose.yml` to `config/`, the volume host path must change from `./uploads` to `../uploads` (relative to the new file location). Flag for T4.

### 5. Dual lockfile concern — both npm and pnpm workspace entries need updating (MEDIUM)

`package-lock.json` contains four hard-coded `"@rw/shared": "file:../shared"` and `"file:../backend"` entries. `pnpm-lock.yaml` contains similar. Both are regenerated by `npm install` / `pnpm install` after the workspace declarations are updated, so no manual edits needed — but **`npm install` must be run after each workspace change in T2 and T3** or the lock files will be stale, breaking CI and anyone running a clean install.

### 6. `backend/vitest.config.ts` has no `root` — T4 move requires one (LOW for T3, HIGH for T4)

The config's `include: ['src/**/*.test.ts']` is implicitly relative to the config file's directory. While the file stays in `packages/api/`, this continues to work. If T4 moves it to `config/api/vitest.config.ts`, vitest will look for `config/api/src/**/*.test.ts` and find nothing. A `root` option pointing back to `packages/api` is mandatory for T4.

### 7. `vite.config.ts` SCSS `resolve(__dirname, 'src/styles/variables.scss')` — fine for T3, breaks in T4 (LOW/MEDIUM)

`__dirname` in `packages/cms/vite.config.ts` = `packages/cms/` → `packages/cms/src/styles/variables.scss` is correct. No change needed in T3. In T4 if the config moves to `config/cms/vite.config.ts`, the same `__dirname` trick resolves to `config/cms/src/styles/variables.scss` (doesn't exist). Needs `path.resolve(__dirname, '../../packages/cms/src/styles/variables.scss')`. Coordinate with T4.

### 8. pnpm `allowBuilds` entries are package-name independent (BENIGN)

`pnpm-workspace.yaml` lists `allowBuilds` for `@parcel/watcher`, `core-js`, `esbuild`, `sharp` — these are third-party package names unaffected by our renames.
