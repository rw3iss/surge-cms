The implementation exposes 26 module namespaces via `createClient` covering all

All prerequisite work was complete before implementation: every route module on
the typed manifest framework, `docs/api-manifest.json` generated from the live
registry, `docs/API.md` documenting all 28 modules / 198 routes, **and every
module shipping request/response DTOs** in `@sitesurge/types` (see below). The
package lives at `packages/cms-client` (`@sitesurge/client`).

## Goal

A typed TypeScript client (`@sitesurge/client`, scaffolded at `packages/cms-client`;
renames with the SiteSurge cut) that mirrors the in-process `cms.*` service
shape over HTTP:

```ts
const cms = createClient({ baseUrl, auth: { apiKey: 'ssk_…' } });
const posts = await cms.posts.list({ status: 'all' });   // typed, paginated
const post  = await cms.posts.getBySlug('hello-world');  // throws ContentLockedError
```

Works in Node and the browser; zero runtime deps (fetch-based).

## Inputs the SDK builds on (all exist today)

| Input | Where | Carries |
|---|---|---|
| `docs/api-manifest.json` | generated by `npm run docs:api` | module, mountPath, per-route method/path/absolutePath/auth/summary |
| `@sitesurge/types` | `packages/shared/src/api/` | `ApiResponse`, `ApiError`, `ErrorCode`, `AuthTier`, `ApiKeyScope`, `ApiMeta`, `ContentLockedDetails` |
| `@sitesurge/types` route DTOs | `packages/shared/src/api/routes/` | request (`<Module><Action>Query`/`Body`/`Params`) + response (`<Module><Action>Response`) DTOs for all 28 modules |
| `@sitesurge/types` entity types | `packages/shared/src/types/` | Post, Page, Campaign, Form, User, MailingList, … |

## Per-module DTOs: COMPLETE

This was the main prerequisite gap; it is now closed. **All 28 modules ship
request/response DTOs** in `packages/shared/src/api/routes/<module>.ts`, each
re-exported via the `packages/shared/src/api/index.ts` barrel. The conventions
are documented in that barrel's header comment:

- Module-prefixed names (`<Module><Action>Query`/`Body`/`Params`/`Response`)
  prevent cross-module collisions.
- List responses type `data` as the element array; pagination rides
  `ApiResponse.meta`.
- Query DTOs describe client-facing input (`page`/`limit` typed `number`
  though they travel as strings).
- DTOs **reference** entity types from `../types/`, never re-declare them; a
  wire-only subset/extension (e.g. `PostWithBlocks`) is defined once here so
  every consumer shares it.

The backend binds its zod schemas to these DTOs (`satisfies z.ZodType<X>`, or an
`AssertCompatible<z.infer<typeof schema>, XQuery>` assertion where coercion makes
input ≠ output), so **server/client drift is a compile error**. The SDK imports
these DTOs directly — no transcription needed.

## Generation strategy (decide at kickoff)

1. **Hand-rolled thin client (recommended start)** — one `request()` core +
   per-module namespaces written by hand against the DTOs. Pros: full control
   over ergonomics, naturally handles the irregular endpoints listed below.
   Cons: manual drift risk (mitigate: a CI check that diffs the SDK's route
   table against `api-manifest.json`).
2. **Manifest-driven codegen** — emit method stubs from `api-manifest.json`
   (method/path/auth are sufficient for signatures-without-types; DTOs imported
   by convention from `@sitesurge/types`). Pros: zero drift. Cons: the irregular
   endpoints need overrides anyway; build tooling to maintain.

## Auth handling

- **API-key mode** (machine clients): `Authorization: Bearer ssk_…`. No CSRF,
  no refresh. Scopes are hierarchical `read < write < admin`; GET/HEAD needs
  read+, mutations write+. Keys cannot call the `/api-keys` management
  endpoints (403 by design) and cannot use `?preview=admin` draft-slug
  previews (JWT-only).
- **User-JWT mode**: `POST /auth/login` → `AuthResponse` (`user`,
  `accessToken` 15 min, `refreshToken` 7 d / 30 d rememberMe) in the body and
  as httpOnly cookies. SDK keeps tokens in memory/storage, sends Bearer,
  auto-refreshes via `POST /auth/refresh` (accepts body token) on 401
  `UNAUTHORIZED` `Token expired` with single-flight de-dupe.
- **CSRF**: Bearer requests (JWT or key) bypass CSRF entirely. Cookie-mode
  state-changing requests need the `csrf-token` cookie + `x-csrf-token` header
  (any GET acquires the cookie). A browser cookie-jar SDK mode must implement
  the round-trip; Bearer mode never needs it. **Login itself is CSRF-guarded**:
  a cookie-less client must GET once (e.g. `/health/live`), read the
  `csrf-token` Set-Cookie, and send it back on the login POST.

## Error mapping

`{ success: false, error: { code: ErrorCode, message, details? } }` everywhere
(except raw endpoints). Map to typed exceptions switchable on `code`:
`UnauthorizedError`, `ForbiddenError`, `NotFoundError`, `ValidationError`
(field details in `details.errors[]`), `ConflictError`, `RateLimitedError`,
`ContentLockedError` (carries `ContentLockedDetails` from `error.details` —
the gated posts/pages preview contract), `InternalError`.

## Pagination & list conventions

- Standard list meta: `{ page, limit, total, totalPages }`.
- **Public vs admin lists**: merged endpoints (posts, campaigns, forms) serve
  the public gate by default; admins/keys opt into the all-statuses view with
  an explicit param — posts: `status`/`sort`; campaigns & forms: `all=true`
  (their public and admin list shapes differ: public = bare array, admin =
  paginated). The SDK should expose these as distinct typed methods
  (`listPublic(): T[]` vs `list(): Paginated<T>`) rather than one overloaded
  call.
- Bulk actions are uniform: `POST /<module>/bulk` `{ ids, action, value? }` →
  `{ updated, action }`.

## Irregular endpoints (collected during the sweep — need SDK special-casing)

| Endpoint | Quirk |
|---|---|
| `PUT /api/v1/settings/` | 409 feature-cascade returns the planner result as `error` (NOT the standard ApiError shape) — needs a dedicated result type |
| `GET /api/v1/search` | `data` is a keyed map (`{posts, pages, campaigns,…}`), `meta` lacks `totalPages` |
| `GET /api/v1/health/detailed`, `/ready` | non-standard envelopes; `success` mirrors health, 503 on degrade — treat as probes |
| `POST /api/v1/setup/install` | responds then restarts the process — connection drops; not idempotent |
| `PUT /api/v1/payments/admin/plans/:id` | union return: `{message:'No changes'}` \| plan object |
| `POST /api/v1/payments/webhook` | raw Stripe contract (`{received:true}`), excluded from SDK |
| `/feed.xml`, `/sitemap.xml`, `/u/:token*`, `/lists/:slug/confirm/:token` | raw XML/HTML outside `/api/v1` — exclude from the typed client |
| `GET /api/v1/audit` | silently clamps out-of-range page/limit (no 400) |
| `POST /api/v1/fonts` | multipart field `file` + `customId`/`familyName` form fields |
| `POST /api/v1/media` / `/block-upload` | multipart `file` / `files[]` (max 10) — SDK needs FormData paths |
| mailing lists | dual-mounted: admin under `/api/v1/mailing-lists`, public subscribe under `/api/v1/lists` — surface as ONE namespace |
| `GET /api/v1/connections` | credentials are masked server-side; never expect raw tokens |

## Deferred API gaps — do NOT expose in the SDK yet

The monorepo gap audit fixed two missing endpoints (`DELETE /users/:id`,
connections reorder) but **deferred two half-built features** pending design.
The SDK should not surface either until they ship:

- **`POST /auth/register`** — public self-registration is not wired (the Join
  flow + role/verification policy needs design). Until then the client exposes
  only login/refresh/logout for auth.
- **`POST /utils/url-preview`** — link-preview/unfurl endpoint is half-built
  (no settled provider/cache/SSRF-guard contract). Omit from the client surface.

When these land they get DTOs in `packages/shared/src/api/routes/` like every
other module, then a client namespace.

## Other contracts worth encoding

- `CONTENT_LOCKED` (posts + pages slug fetch): 403, `error.details =
  { locked, accessLevel, preview: { title, description, featuredImage } }`
  (description/featuredImage nullable).
- Key-authored writes set `author_id`/`created_by` etc. to NULL (synthetic
  actors); audit history shows `api-key:<name>` in `new_values.actor`.
- `social` / `socialFeed` services are not part of the `cms.*` aggregate —
  decide whether the HTTP SDK mirrors the aggregate or the manifest (recommend
  the manifest: it is the wire truth).
- Versioning: the API is served under `/api/v1`; recommend the SDK pins a
  manifest `generatedAt`/hash and warns on drift. Define a breaking-change
  policy before publishing (none exists yet).

## Open questions — RESOLVED

1. **Publish target:** monorepo workspace (`private: true`), npm-publish-ready
   structure (exports map, `files: ["dist"]`, dual ESM+CJS+.d.ts) — publish when
   the SiteSurge rename lands.
2. **Hand-rolled vs codegen:** hand-rolled thin client. One `request()` core +
   26 per-module namespaces written against the DTOs. CI drift guard via
   `npm run check:drift -w packages/cms-client`.
3. **Cookie-mode in v1:** Bearer-first (localStorage persistence, auto-refresh).
   Cookie mode is supported as an option but Bearer + API-key are the primary paths.
4. **Scope rights:** API-key scopes (`read < write < admin`) encoded in the
   client; `cms.apiKeys` rejects key callers at the server (403 by design).
5. **Retry / idempotency:** GET auto-retry (3 attempts, exponential backoff);
   writes opt-in via `MutationOptions.retry`. Forward-compat `idempotencyKey`
   field on `MutationOptions`.
